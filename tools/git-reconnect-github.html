كيف تربط مشروعك المحلي بمستودع GitHub من جديد

دليل شامل لإعادة ربط مشروعك المحلي بمستودع GitHub، مع شرح المسارات المختلفة والمشاكل الشائعة وحلولها.

2026-05-01

واحدة من أكثر المشاكل المزعجة التي تواجه المطورين — سواء المبتدئين أو حتى من لديهم خبرة — هي فقدان مجلد الإعدادات .git داخل مشروعهم المحلي، أو محاولة ربط مجلد جديد بمستودع قديم موجود مسبقاً على GitHub. في هذا الدليل، سنتعلم معاً كيف نعيد هذا الارتباط بخطوات واضحة، ونختار المسار الصحيح بحسب حاجتنا.

لماذا تحدث هذه المشكلة؟

عندما تعمل مع Git، يُخزَّن كل تاريخ المشروع وإعداداته داخل مجلد مخفي اسمه .git. إذا حُذف هذا المجلد أو بدأت مشروعاً جديداً من الصفر دون تهيئة Git، يفقد الجهاز أي معرفة بـ GitHub ويصبح كل شيء غير مرتبط. والنتيجة: لا يمكنك رفع تعديلاتك ولا متابعة سجل العمل.

الخطوة الأولى: تهيئة المشروع وربطه

قبل البدء، يجب أن نتأكد أن المجلد المحلي مُهيَّأ كمستودع Git ومرتبط بالرابط الصحيح. افتح الـ Terminal داخل VS Code ونفّذ الأمرين التاليين:

أولاً، هيّئ المستودع محلياً:

git init

ثانياً، اربط المجلد برابط GitHub الخاص بك (استبدل الرابط برابط مستودعك):

git remote add origin https://github.com/username/repository.git

هذان الأمران كافيان لإنشاء الجسر بين جهازك والمستودع البعيد. لكن الخطوة التالية تختلف بحسب ما تريد تحقيقه.

الخطوة الثانية: اختر مسارك

هنا يتفرق الطريق. اقرأ الخيارين جيداً واختر ما يناسب وضعك:

المسار (أ): الرفع القسري — الأسرع والأنظف

اختر هذا المسار إذا كنت تريد أن تكون النسخة الموجودة على جهازك هي النسخة الوحيدة والأساسية، أو إذا كنت لا تهتم بالملفات القديمة على GitHub وتريد استبدالها بالكامل. الأوامر: 

git add .
git commit -m "تحديث الملفات واستبدال النسخة القديمة"
git push origin main --force
"تذكّر: الرفع القسري سيحذف أي ملفات موجودة على GitHub وليست موجودة في جهازك حالياً. استخدمه بوعي."
— نصيحة مهمة قبل التنفيذ

المسار (ب): الدمج مع الحفاظ على التاريخ

اختر هذا المسار إذا كان المستودع على GitHub يحتوي على ملفات مهمة مثل README أو ملفات التراخيص، أو إذا كنت تريد دمج تعديلاتك الجديدة مع ما رُفع سابقاً. نفّذ أولاً: 

git pull origin main --allow-unrelated-histories

الخيار --allow-unrelated-histories ضروري هنا لأن المجلد المحلي لا يملك تاريخاً مشتركاً مع المستودع البعيد، وبدونه سيرفض Git عملية الدمج.

بعد اكتمال الدمج (أو حل أي تعارضات)، أكمل بهذه الأوامر:

git add .
git commit -m "دمج الملفات المحلية مع ملفات GitHub"
git push origin main

التعامل مع تعارضات الدمج (Conflicts)

إذا ظهرت رسالة تعارض بعد أمر pull، لا تقلق — هذا طبيعي جداً. افتح الملفات التي فيها تعارض في VS Code وستجد علامات مثل <<<<<<< و>>>>>>> تفصل بين النسختين. احذف الجزء الذي لا تريده وأبقِ على ما يناسبك، ثم احفظ الملف ونفّذ أوامر الـ commit والـ push كما هو موضح أعلاه.

مشاكل شائعة وحلولها

خطأ: Remote origin already exists

يظهر هذا الخطأ عندما يكون المجلد مرتبطاً بالفعل بمستودع آخر. بدلاً من إضافة رابط جديد، غيّر الرابط الحالي بهذا الأمر: 

git remote set-url origin https://github.com/username/repository.git

خطأ: Rejected — non-fast-forward

هذا الخطأ يعني أن GitHub يرفض استقبال ملفاتك لأن المستودع البعيد يحتوي على تغييرات أحدث مما لديك. الحل إما استخدام المسار (أ) للرفع القسري إذا لم تهتم بتلك التغييرات، أو المسار (ب) لجلب التغييرات ودمجها أولاً.

نصيحة ذهبية قبل أي خطوة

قبل تنفيذ أي أمر كبير، نفّذ دائماً: 

git status

هذا الأمر البسيط يمنحك رؤية واضحة وشاملة: ما الملفات التي ستُرفع؟ ما الملفات الجديدة؟ ما الذي تغيّر؟ دقيقة واحدة في قراءة ناتج هذا الأمر يمكن أن تنقذك من أخطاء كبيرة.

الخلاصة

إعادة ربط مشروع بـ GitHub ليست بالمعقدة إذا فهمت الخطوات بشكل صحيح. ابدأ دائماً بـ git init وgit remote add origin، ثم اختر مسارك بناءً على ما تريد: الرفع القسري إذا أردت السيطرة الكاملة، أو الدمج إذا أردت الحفاظ على الملفات القديمة. والأهم: لا تتسرع — اقرأ رسائل الأخطاء بتمعن فهي دائماً تحمل الحل.

إذا واجهت مشكلة لم تجد لها حلاً هنا، تواصل معي وسأحاول مساعدتك!
